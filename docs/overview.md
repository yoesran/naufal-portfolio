# Overview & Architecture

> Technical source of truth for the current code. See [handoff.md](./handoff.md) for the planning context and _why_ behind these choices.

## What this is

A portfolio where **the architecture is the portfolio** — a microfrontend system demonstrating cross-framework Module Federation, connecting three independently deployable apps:

- **`naufal-host`** (React) — composes federated components at runtime.
- **`naufal-lab`** (Svelte) — a remote that exposes components over Module Federation.
- **`naufal-party`** (PartyKit) — a WebSocket server powering multiplayer presence.

Federation uses `@module-federation/vite` — the actively maintained, framework-agnostic plugin. (The older `@module-federation/nextjs-mf` is being deprecated and was deliberately not used.)

The remote exposes **two** components today: `./SpringToy` (a draggable Svelte "lanyard ticket" you mount into — and tug out of — the React host) and `./Presence` (a page-wide multiplayer cursor overlay that holds its own live WebSocket). Presence is the on-message demo: a federated Svelte component, embedded in a React host, sharing realtime state across deployments — surfaced as a global, opt-in cursor overlay toggled from the host header.

The blog (`naufal-blog`) is a standalone static site, **internationalized (EN/ID) with locale-routed URLs** (`/en`, `/id`) for proper multilingual SEO (hreflang + per-locale `sitemap`) — a branded home, the CV page (data + content translated), and an MDX-driven `posts` section (`@next/mdx` + `remark-gfm`, a `[lang]/posts/[slug]` route prerendered per locale × slug, with one `.mdx` body per language). It has a **light/dark/system theme toggle** and a per-post **reading panel** (font family / size / background), both no-FOUC. A root `app/layout.tsx` owns `<html>`/`<body>` above the `[lang]` segment (so a locale switch never re-renders them — keeping the theme class and avoiding a React-19 script warning, see [gotchas.md](./gotchas.md) #24); unmatched URLs use a standard `app/not-found.tsx`. The locale-less section paths (`/cv`, `/posts`) are Cloudflare `_redirects` rules to the default locale, but the apex `/` serves a client stub that **detects the visitor's browser language** and routes to `/en` or `/id` (a static export has no edge to read Accept-Language; no middleware either). It's **deployed at `naufal-blog.pages.dev`** (its own Cloudflare Pages project, independent of the federated three); two posts — `styling-across-module-federation` and `writing-with-mdx` — are live in both locales.

**`badriyatim-family`** is a separate product that shares the repo but nothing else: a private family app (silsilah/events/iuran for Naufal's extended family), deliberately **not** federated and outside the portfolio's design system — it has its own ("Songket & Rumah Gadang", Bahasa-only). It documents itself in `badriyatim-family/CLAUDE.md` + `README.md`; the one repo-level rule worth knowing is its **PII data split**: public pages import only `data/public.json` (names/tree), while the full seed `family.json` is kept **outside the repo entirely** (it goes straight into Supabase; the `data/family.json` gitignore entry remains as a tripwire) — the privacy guarantee is that split, which is what lets this repo be public with the app in it.

**`nuansa`** is the other in-repo product: a platform for the sites that exist only to display information a client hands over — online invitations first, then linktree-style profiles, menus, event guides. Also standalone, also outside the portfolio's design system, also Bahasa-only. Its architecture is the point: a **section** is a content contract (Zod-backed editable fields, zero styling), a **shell** is a visual identity (chrome plus one renderer per section it supports), and a **template** is just a preset naming a shell and a section order. A shell renders exactly the sections it wrote a renderer for, and that absence _is_ the compatibility rule — which is what stops two templates from being the same page reskinned. It documents itself in `nuansa/SPEC.md` (the architecture), plus its own `CLAUDE.md` + `README.md`. One snag worth knowing before you run it: `nuansa` and `badriyatim-family` both take Next's default **:3000**, so start one with an explicit port (`npm run dev -- -p 3005`) if you want both up at once.

**`sukamotret`** is the first client piece: a one-page landing site for a real photo studio in Tabalong, South Kalimantan — built as freelance presence (a live site for a real business, with a measurable outcome). The concept is **light**: today's golden/blue hours computed in the browser from the studio's real coordinates (NOAA solar math, no API), a day-scrubber that relights a canvas scene from real sun altitude, a before/after photo editor (film-look grades + intensity, never uploads the visitor's photo), a pictogram **pose guide** (the studio's real value is direction), and a session builder that composes a WhatsApp brief ending in _"Dari website."_ — the lead-counting mechanism. Bahasa-only, static export, Tailwind v4 + shadcn (base-vega, nuansa's setup) with the studio's own palette mapped onto the semantic tokens. Everything is **dummy data** until the studio confirms facts — the launch checklist lives in `sukamotret/README.md`, the gotchas in its `CLAUDE.md`. **Deployed as a draft at `sukamotret.pages.dev`** (own Cloudflare Pages project, direct upload like the siblings; the visible Draft banner and a deliberately fake WhatsApp number make it safe to share before launch). It also claims **:3000** for dev and its Playwright suite, so it can't run alongside the other two Next apps.

**`kopilima`** is the second client piece: a one-page site for **Kopi Lima**, a real no-franchise coffee chain (five branches across Balikpapan and Samarinda, three open 24 hours, everything Rp15.000). The difference from `sukamotret` is that **the brand already exists** — the identity was extracted from their live Instagram (electric ultramarine `#3300f8`, amber checkered flags, "Daily Cruisin'", the Coffee-Mechanic vocabulary) and the page follows it rather than inventing one: the ground _is_ the brand blue, and every informational surface sits on it as a cream **paper sticker** (ink frame, hard offset shadow, sharp corners, slight tilt). Plus a draggable **gerobak** that rolls through the branch-opening timeline, a menu where tapping a drink **pours its layers into a cup**, a table-preference floor plan, an ephemeral **cart that composes a struk** you show the cashier (no storage, no backend — the UI says so), and a bulk calculator where the flat 15k price makes the total one visible multiplication. Cities are **never mixed** (Balikpapan and Samarinda are separate worlds), and every composed brief ends in the code **`WEB-5`** — the lead counter, since Instagram DMs can't be prefilled the way `wa.me` links can. Bahasa-only, static export, same stack as `sukamotret`. Everything the owner hasn't confirmed is **dummy data** behind a Draft banner — the launch checklist is in `kopilima/README.md`, the blue-wall design system and its gotchas in its `CLAUDE.md`. **Deployed as a draft at `kopilima.pages.dev`** (own Cloudflare Pages project, direct upload like the siblings). ## Key design decisions

- **Mount-function pattern** over exposing components directly: framework-agnostic, keeps the Svelte runtime out of the host bundle. Reused for every remote regardless of framework. See [mf-core.md](./mf-core.md).
- **Generic `RemoteMount`** over per-remote wrappers: one component handles every remote, plus load status + graceful fallback.
- **`opts` carries context** across the boundary as a plain object — the forward-compatible hook (originally floated for `locale`).
- **Realtime inside a microfrontend**: `Presence` proves a federated remote can own a live WebSocket and share state across deployments. See [features.md](./features.md).
- **Ship CSS with the exposed entry** so Tailwind works embedded (theme via CSS vars, utilities via shipped stylesheet). See [mf-platform.md](./mf-platform.md).
- **No shared Svelte**: `shared: []` — Svelte is fully bundled into the remote. Host shares only `react` / `react-dom`.
- **`build.target: 'chrome89'`** on both sides: required by `@module-federation/vite` (native ES modules + top-level `await`).
- **`127.0.0.1` everywhere**: avoids the Windows IPv6 / dts-plugin IPv4 mismatch. See [gotchas.md](./gotchas.md).
- **React Compiler via a ~12-line custom plugin**, not `@rolldown/plugin-babel`. The documented integration is silently inert in this project (federation + plugin-react v6 interaction); see [features.md](./features.md).
- **i18n is per-remote, locale crosses via `opts.locale`**. Host runs `i18next` + `react-i18next`; each remote will own its own library when translated. Host sets `<html lang>`; remotes will read locale from their mount's `opts`. See [features.md](./features.md).
- **Theming lives on the host, not in any remote**. The theme drawer (a header slide-over `Sheet`) makes the host customizable across five axes (mode, accent, surface, radius, font) — all values on the host's `<html>` (class / `data-*` / CSS vars), backed by one shared store. Because they're CSS variables, they cascade into every remote's mounted DOM with zero coordination code, so re-skinning the page re-skins the federated remote live — a working demo of the cross-boundary cascade. See [features.md](./features.md).
- **Scroll-reveal is plain `IntersectionObserver` + CSS transition**, not Framer Motion (or any animation library). The same hook also backs `RemoteMount`'s optional viewport gate for federated `load()` — though both current remotes opt out with `eager` and load on an explicit user action instead (the live-remote **Run**, the presence toggle). See [features.md](./features.md).
