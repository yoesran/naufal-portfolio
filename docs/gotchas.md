# Gotchas Already Solved (don't re-discover)

The single consolidated list — previously duplicated across the MF-setup and handoff docs (which let them drift), now one copy. Also the raw material for the blog posts — the Tailwind-across-the-boundary lesson has already shipped as one; the MF build, the TS6/baseUrl trap, and the eager-preload / `loadRemote` pattern remain strong candidates.

Cross-references point at [mf-core.md](./mf-core.md), [mf-platform.md](./mf-platform.md), and [features.md](./features.md) for the full treatment.

1. **Svelte 5 `new Component()` is gone** — components are functions, not classes. The Svelte 4 API produces misleading `Cannot read properties of null (reading 'nodes')` errors. Use `mount()` / `unmount()`, wrapped in a mount adapter. See [mf-core.md](./mf-core.md) §1.

2. **Windows + dts + `localhost` = silent failure** — the dts-plugin forces IPv4; Windows resolves `localhost` to IPv6 (`::1`) → "Failed to download types archive". Use `127.0.0.1` everywhere in the federation `entry` and the remote's `server.host` / `preview.host`. See [mf-core.md](./mf-core.md) §4.

3. **Remote runs with `vite build --watch` + `vite preview`, not `vite dev`** — `@module-federation/vite` generates `remoteEntry.js` at build time; there's no `remoteEntry` in dev mode.

4. **No HMR across the boundary** — a host plugin watches `../naufal-lab/dist/` and sends `{ type: 'full-reload' }` over Vite's WebSocket. Requires `server.fs.allow` to permit watching a sibling folder. See [running-locally.md](./running-locally.md).

5. **React Strict Mode double-mounts** the remote (mount → unmount → remount to verify cleanup). Intentional; the single counter seen is the second mount. Don't disable Strict Mode.

6. **`@mf-types/` is gitignored** — a regenerated artifact like `node_modules`/`dist`. First run on a fresh clone shows "cannot find module" until the host fetches once. See [running-locally.md](./running-locally.md).

7. **TS 6.0 `baseUrl` is a hard error (TS5101)** that silently kills DTS generation through the dts-plugin's inherited tsconfig — the host gets no types, with only a `#TYPE-001` in the build log. (shadcn-svelte's init added the `baseUrl`.) Add `ignoreDeprecations: "6.0"` (band-aid; removed entirely in TS 7) or drop `baseUrl`. See [mf-core.md](./mf-core.md) §5.

8. **`vite build --watch` does NOT reload `vite.config.ts`** — unlike `vite dev`, it watches source but not the config. After changing `exposes` (e.g. adding `./Presence`), restart the build process or the change never reaches `remoteEntry.js`.

9. **Tailwind utility classes don't cross the MF boundary** — theme **CSS variables** cascade from host into the embedded remote (shadcn tokens just work), but utility classes (`bg-sky-400`, the shadcn Button classes) live in the remote's entry stylesheet, which the host never loads. Import the stylesheet **inside the `.svelte` component** (not the `.ts` mount adapter — see #10) so it ships with the chunk, or style remotes with CSS variables + inline only. See [mf-platform.md](./mf-platform.md).

10. **DTS-generated tsconfig only extends the root `tsconfig.json`**, not `tsconfig.app.json`. Anything app-only (`vite/client` types, svelte types, strict settings) is invisible to type generation — so a bare side-effect `import '../app.css'` in a `.ts` mount adapter fails with `TS2882`. Put the import in the `.svelte` file, or add the needed types to the root config. Same family for `svelte-i18n`: its setup module imports JSON dictionaries, so set `resolveJsonModule` on the **root** config (not just `tsconfig.app.json`) and import that module from the `.svelte` components, not the `.ts` adapters — keeps the dts compile away from the JSON entirely.

11. **The plugin eagerly preloads every `import('lab/X')` literal it finds**, blocking `main.tsx` until all of them are fetched (the "blank screen for a second" over a throttled link). Use `loadRemote('lab/X')` from `@module-federation/runtime` for block-level loads to opt out — only the small `remoteEntry.js` then blocks bootstrap, chunks load lazily after React mounts. **The "only remoteEntry.js" residue turned out to be ~1s of deployed FCP itself** (a cross-origin fetch mid-bootstrap — or a 3s connection timeout locally with the lab down), so the lab was later removed from build-config `remotes` entirely and is registered at runtime on first use; the serial MF bootstrap chain is flattened with injected `modulepreload` hints. FCP 4.4s → ~1s. See [mf-platform.md](./mf-platform.md) › Initial-load performance.

12. **`hostInitInjectLocation` already defaults to `'html'`** — setting it explicitly is a no-op. The blank-on-startup problem isn't this option; it's the eager-preload behaviour (#11).

13. **MF plugin rewrites `import('...')` literals even inside comments.** Writing `import('lab/SpringToy')` in a `.ts` source comment makes Vite's import analysis try to transform it, breaking the file with `Failed to parse source for import analysis`. Refer to federated imports in prose as just `lab/SpringToy` (no `import()` wrapper).

14. **First-paint white flash isn't MF** — even with everything fully lazy, the browser paints `index.html` before the JS bundle downloads/parses/executes. Mitigated by an inline `<style>` in `<head>` setting `background-color` + `color-scheme` per `html` / `html.dark`, plus an inline `<script>` that resolves the user's stored theme + system preference and sets the class **before** the style is parsed (so the first frame is the right theme, not just _a_ theme). Both are **generated from `theme-tokens.ts`** by the `theme-prepaint` Vite plugin (which replaces a `<!-- theme-prepaint -->` slot in `index.html`), so the pre-paint logic shares the store's single source of truth instead of being a hand-copied second implementation that silently drifts. The tokens module is intentionally DOM-free so it imports cleanly under both the app and the (no-DOM) `vite.config.ts` tsconfig. See [features.md](./features.md).

15. **Browsers fire `touchcancel` (not `touchend`) when they promote a touch to a scroll/zoom** (or the OS interrupts). Any touch-driven state without a `touchcancel` handler can leak — finger-down refs stuck `true`, animations stuck paused, etc. Always handle `touchcancel` alongside `touchend` for touch UI. (`TechStackBlock`'s orbit pills do exactly this.)

16. **`@rolldown/plugin-babel` + `reactCompilerPreset` silently does nothing in this project.** The documented Vite 8 / plugin-react v6 path for the React Compiler builds cleanly and reports no errors but never runs the babel transform on any source file (likely an interaction with `@module-federation/vite`'s environment handling, not pinned down). Verify by setting `panicThreshold: 'all_errors'` — if the build still succeeds on any non-trivial source, babel isn't running. Replace with a ~12-line custom Vite plugin calling `@babel/core` directly. See [features.md](./features.md).

17. **iOS Safari mobile-tap on a non-focusable element doesn't blur a focused button.** Patterns built on the button's `onBlur` to detect "tapped somewhere else" (to deselect, close a popover, dismiss state) silently fail on iOS — focus stays on the previously-tapped button, blur never fires, the stale state persists. Workaround: register a document-level `touchstart` listener and run the deselect when the event target **isn't one of your interactive children**. Match against a shared marker on those children (a `data-` attribute — `TechStackBlock` uses `!target.closest('[data-orbit-slot]')`, the same marker its `onBlur` checks) rather than the container's bounds: that also deselects on taps that land on inert areas _inside_ the container, and lets the children own their own selection via `onTouchStart`. Use **capture phase** (`{ capture: true }`) so descendants calling `stopPropagation` (base-ui-react's Menu trigger does this) can't swallow it — ordering stays correct because the marker test identifies child taps regardless of which listener fires first. Also force-reset any "interaction in progress" refs in the same handler — multi-touch and scroll-promotion sequences can leave them stuck `true`. See [TechStackBlock.tsx](../naufal-host/src/components/blocks/TechStackBlock.tsx).

18. **React Compiler memoizes `<Trans>`, so it doesn't re-translate on a locale switch.** With `babel-plugin-react-compiler` on, a `<Trans i18nKey="…" components={{…}}>` whose props don't change across a language switch (constant key + a compiler-memoized `components` object) gets its element cached — so plain `{t('key')}` updates on toggle but the `<Trans>` block stays in the old language. Fix: pass the live `t` from `useTranslation()` as `<Trans t={t} …>`. react-i18next returns a **new `t` reference** on `languageChanged` (which is also why plain `t()` keeps working under the compiler), giving the element a changing input so it re-renders translated. Corollary: any block containing a `<Trans>` must call `useTranslation()` itself to subscribe — don't rely on a parent re-render, since the compiler memoizes children too (this bites any block that renders a `<Trans>` but forgets its own `useTranslation`). See [features.md](./features.md).

19. **A remote's injected CSS can override host theming via cascade ties.** The remote ships its own `app.css` (Tailwind preflight + a global `:root {…}` token block + `@layer base html {…}`), and the MF runtime injects it into the **host document** when the remote loads. Those globals collide with the host's own at **equal specificity**, and since the remote injects _after_ the host bundle, the remote wins the tie. This is invisible while host and remote agree on values (the default tokens), but the moment the host diverges — e.g. a theme customizer setting `[data-surface=x]{--background}` (0,1,0) or `html{font-family}` (0,0,1) — the remote's `:root{--background}` / `html{font-family:Inter}` silently override it once a remote chunk mounts. Fix by making the host's theming rules out-specify the remote's injected globals: `:root[data-surface=x]` (0,2,0 beats `:root`), and put the host font rule on `:root` not `html` (0,1,0 beats `html`). Accent (`--brand`, never redefined by the remote) and radius (applied as an **inline** style on `<html>`, which beats any stylesheet) are safe by construction. Note this only manifests at runtime after the remote injects — static typecheck/build won't catch it. See [features.md](./features.md).

20. **A portaled popup can't be hidden with a responsive class on its trigger.** base-ui's `Menu` (our `DropdownMenu`) renders its popup through a `<Portal>` into `document.body`, so it is **not** a DOM descendant of the trigger. Putting `sm:hidden` on the trigger (the mobile-nav menu button) hides the button on desktop but leaves the `<Menu.Root>` live, and the portaled popup is outside the `sm:hidden` subtree — so once anything opens it, it renders anyway, anchored to the now-zero-box (display:none) trigger, which floating-ui positions at the viewport origin (i.e. under the sticky header). CSS can't fix this from the trigger side. Fix: **unmount the whole menu on desktop in JS**, not via CSS — `Header.tsx` renders the `DropdownMenu` only when `!useMediaQuery('(min-width: 640px)')`. (naufal-blog's [`MobileMenu`](../naufal-blog/src/components/MobileMenu.tsx) drawer does the identical JS-unmount for the blog's phone menu, but with its own **SSR-safe** [`useMediaQuery`](../naufal-blog/src/lib/useMediaQuery.ts) that starts `false` so the static-export prerender and the client's first render agree — reading `matchMedia` eagerly like the host's hook would trip a hydration mismatch.) General rule: anything that portals to `<body>` (menus, tooltips, dialogs) must be gated by a JS media-query check, never a Tailwind `sm:`/`md:` class on an ancestor. The shared [`useMediaQuery`](../naufal-host/src/lib/useMediaQuery.ts) hook is the single place for JS viewport/device checks (also used for `TechStackBlock`'s `(hover: hover) and (pointer: fine)` desktop detection and the reduced-motion checks).

21. **Cloudflare Pages `_headers` merges overlapping rules — duplicate `Access-Control-Allow-Origin` breaks CORS.** The MF remote needs CORS so the cross-origin host can fetch `remoteEntry.js` + chunks. Setting `Access-Control-Allow-Origin: *` in _both_ a broad `/*` rule and a specific `/remoteEntry.js` rule doesn't "override" — Pages **combines** the matching rules and emits `Access-Control-Allow-Origin: *, *`, which is invalid (the value must be a single origin or one `*`), so the browser rejects every remote fetch with a CORS error. Confirmed by `curl -I` on the deployed file showing the doubled value. Fix: set ACAO **once** (the broad rule), and put only non-CORS headers (e.g. `Cache-Control: no-store` on the manifest) in the specific rule. See [deployment.md](./deployment.md).

22. **A federated remote is always a production build, so it can't use Vite's dev/prod env split — and its standalone page needs its own backend config.** Two linked traps, both surfaced deploying the realtime presence remote. (a) The host distinguishes dev/prod via `vite` (serve — dev defaults, or an optional `.env.local`) vs `vite build` (loads `.env.production`); the remote is built in _both_ cases (`vite build --watch` for dev, `vite build` for deploy) → both are production mode → the dev watcher would pull prod env. Run the dev watcher with **`--mode development`** so local dev gets dev env (e.g. the standalone page talks to the local PartyKit, not prod). (b) When embedded, the remote receives backend config from the host via `opts`; its **standalone page has no host to pass it**, so if `App.svelte` mounts the component without supplying e.g. the party `host`, it silently falls back to the component's `127.0.0.1:1999` default and the _deployed_ standalone connects to localhost. Fix: the standalone entry reads its own `VITE_PARTY_HOST` from `.env.production` and passes it. Corollary: `VITE_` vars are **baked at build time** — after editing `.env.production` you must rebuild before redeploying, or the old value (or the localhost fallback) ships. See [deployment.md](./deployment.md).

23. **A print layout tuned to fit one page overflows to a second page on devices missing the design font.** The blog's CV is a pixel-faithful replica sized to fill exactly one A4 page using **Calibri**'s compact metrics. On Android (and Linux), neither Calibri nor its named fallback Carlito exists, so the browser substitutes Roboto — whose taller metrics render the same content at ~1136px against the ~1123px (297mm @ 96dpi) page box, spilling a near-empty second page. Two-part fix: **(a) bundle [Carlito](https://fontsource.org/fonts/carlito)** (`@fontsource/carlito`, Google's _metric-compatible_ Calibri clone — identical advance widths + line metrics), so every non-Windows device renders identically to Windows (stays pixel-faithful **and** one page); **(b) in `@media print`, pin the sheet to a fixed `794px` (A4 @ 96dpi) width instead of `100%`**, so line-wrapping — and thus the tuned height — is deterministic across print engines rather than depending on the device's print viewport. Verify by emulating print media and measuring the sheet's `getBoundingClientRect().height` against 1123px under different font stacks (Calibri/Carlito ≈ 1089px ✓; Arial/Roboto ≈ 1136px ✗). See `naufal-blog/src/app/[lang]/cv/`.

24. **Putting `<html>` + a `<script>` in a layout that re-renders on navigation breaks the theme _and_ warns (React 19).** The blog first rendered `<html>`, the no-FOUC pre-paint `<script>`, and the theme class in `[lang]/layout` — which re-renders on **every locale switch**. Two bugs followed: (a) on the switch React re-applied `<html>`'s rendered `className`, wiping the `.dark` class the toggle/prepaint had set imperatively → **the theme reverted when changing language**; (b) React 19 warned _"Encountered a script tag while rendering React component… never executed when rendering on the client"_ because the script element was re-rendered during the (client) navigation — and this fires for an inline `<script>`, **and** for `next/script` with inline children, **and even with `src`** while it lives in a re-rendering layout. The real fix is structural: a **root `app/layout.tsx` _above_ `[lang]`** owns `<html>`/`<body>` + the `beforeInteractive` script (external `public/prepaint.js`); `[lang]/layout` is reduced to nested chrome. The root layout doesn't re-render on a locale switch, so the class survives and the script isn't re-rendered. Verify by driving the **production** export with Playwright (toggle dark → switch locale → assert `.dark` persists, console clean); for `dev`, use **`localhost`** not `127.0.0.1` (the latter breaks Playwright's dev HMR/hydration, so client interactivity silently no-ops and you'll wrongly conclude "fixed"). The warning is **dev-only** and fires on _client_ render, so reproduce by navigating a hydrated page, not by loading.

25. **Static-export i18n: keep a real root layout above `[lang]`.** App-Router i18n nests routes under `app/[lang]/` with `generateStaticParams` returning the locales (supported with `output: 'export'` — the "no i18n" caveat is Pages-Router only). Keep `app/layout.tsx` as the **root** (`<html>`/`<body>`, stable across locale nav — see #24) and make `[lang]/layout.tsx` a **nested** layout (chrome + per-locale `generateMetadata`). Consequences: (a) a standard `app/not-found.tsx` composes fine (no need for experimental `globalNotFound`), with `[lang]/not-found.tsx` for in-locale `notFound()`; (b) middleware/`redirect()` don't run in a static export, so the locale-less **section** paths (`/cv`, `/posts`, so external links stay locale-agnostic) → default locale via Cloudflare `public/_redirects`, while the **apex `/`** is deliberately not an edge redirect — it serves `app/page.tsx`, a client stub that reads `navigator.languages` and routes to `/en` or `/id` (the edge can't read Accept-Language in a static export), with a `<noscript>` meta-refresh to the default locale for crawlers/JS-off; (c) the root `<html>` can't read the locale param, so set `<html lang>` client-side in `prepaint.js` from the URL — the per-page hreflang in metadata carries the SEO signal. See `naufal-blog/src/app/`.

26. **Next 16's default segment prefetching 404s under `output: export`.** Next 16's routing overhaul ships **incremental "segment" prefetching** by default (the v16 upgrade notes: _"you may see more individual prefetch requests"_). Each `<Link>` near the viewport prefetches **per-segment** RSC payloads, requesting URLs like `/en/__next.$d$lang.__PAGE__.txt` (`$d$lang` = the `[lang]` dynamic segment). But the static export emits those segment files at a different path **shape** — `/en/__next.$d$lang/__PAGE__.txt` (a directory + slash, not a flat dotted name) — so every prefetch 404s on a static host. Confirmed against `out/`: the requested flat file is absent, the slash-path file exists; it 404s on Cloudflare Pages too, not just a local server. It's benign (click navigation falls back to fetching the real page, which works) but spams console/network with 404s, and there is **no config flag to disable segment prefetch** in Next 16 (the docs only offer `prefetch={false}` per-link). Fix: a thin `next/link` wrapper ([`naufal-blog/src/components/Link.tsx`](../naufal-blog/src/components/Link.tsx)) that defaults `prefetch={false}` and that **all** internal links import instead of `next/link` — prefetch can't work in this export regardless, so disabling it is pure win (failed requests vanish, navigation unchanged). `forwardRef` through to the underlying `<a>` so Base UI's menu (which clones the link via its `render` prop) keeps the ref it needs for keyboard focus. Importing `next/link` directly anywhere reintroduces the 404s. Verify by driving the **production** export with Playwright and asserting zero `404` responses (and that menu `ArrowDown` still focuses an item — proves the ref survived). Per the blog's `AGENTS.md`, this was diagnosed against the bundled `node_modules/next/dist/docs`, not training data.

27. **Rendering a navigation link through Base UI's `Dialog.Close` (or any button-role primitive) forces button semantics and warns.** Base UI components that act as buttons default to `nativeButton: true`. Passing `render={<Link/>}` (an `<a>`) — e.g. to make a drawer nav link _also_ close the drawer — fires a dev console warning ("_A component that acts as a button expected a native `<button>`… Rendering a non-`<button>` removes native button semantics_") and is semantically wrong: a nav link should be `role=link`, not a button. Two fixes: set **`nativeButton={false}`** to keep the anchor (fine for a non-nav close affordance), or — better for a link that also dismisses an overlay — make the overlay **controlled** (`open` / `onOpenChange`) and close it from the link's `onClick`, leaving the `<Link>` a plain anchor. `naufal-blog`'s [`MobileMenu`](../naufal-blog/src/components/MobileMenu.tsx) drawer does the latter: nav links stay `role=link` and call `setOpen(false)`; only the `×` (a real button) uses `Sheet.Close`. **Catch it in `next dev`, not the static export:** this is a **dev-only** warning (stripped from production builds), so a Playwright pass over `out/` won't surface it — drive `next dev` on `localhost` (see #24) and read the console. The same is true for React's hydration/`<script>` warnings — production-export checks give false confidence for the whole class of dev-only diagnostics.

28. **CSS `zoom` scales inconsistently on iOS Safari — use `transform: scale()` to fit a fixed-size document.** The CV is a fixed **794px** (A4 @ 96dpi) sheet scaled down to fit narrow viewports. With `zoom: var(--cv-zoom)` it fit correctly on Chrome/Android but rendered **~1.7× too tall on iOS Safari** — WebKit scales the box width but not the text/line-layout uniformly, so the CV looked oversized and overflowed (the original "font is wrong on iOS" report — it was sizing, not the typeface). Confirmed by measuring `.cv-sheet` `getBoundingClientRect()` in Playwright's **WebKit** (Safari's engine, installable via `npx playwright install webkit`): Chromium `343×471` vs WebKit `358×822` for the same content; after the fix both match (`~358×491`). Fix: scale with **`transform: scale(var(--cv-zoom))`** (uniform on every engine) instead of `zoom`. Because `transform` does **not** reflow (the layout box keeps its full 794px size), three things are needed: (a) the wrapper's height is set in JS to the sheet's _scaled_ height (`wrap.style.height = sheet.getBoundingClientRect().height + 'px'`) or a large gap is left below; (b) `overflow: hidden` on the wrapper clips the now-overflowing full-size layout box (the visual is within bounds); (c) a width-guard in the `ResizeObserver` callback (`if (width === lastWidth) return`) so mutating the wrapper height doesn't re-trigger the observer into a loop. `@media print` resets `transform: none` + wrapper `height: auto`/`overflow: visible` so the full sheet still prints one A4 page (see #23). General rule: to diagnose an iOS-Safari-only rendering bug, drive Playwright **WebKit**, not Chromium — it reproduced this exactly. See `naufal-blog/src/app/[lang]/cv/`.

29. **A failed federated load can't be retried by re-calling `loadRemote` — the browser caches the failed `import()` by URL.** The remote is `type: 'module'`, so the MF runtime fetches `remoteEntry.js` with a native dynamic `import()` — and browsers cache a **failed** module fetch in the page's module map: every later `import()` of the same URL rejects instantly with the cached error, **no network request** (verified in the live-remote block: after a real failure, Retry produced zero requests even with the remote back up; only a page reload recovered). Neither re-running `loadRemote` nor `registerRemotes(..., { force: true })` alone fixes it — `force` clears the MF runtime's own entry/module caches but can't touch the browser's module map. Fix: on retry, force-re-register the remote with a **cache-busted entry URL** (`remoteEntry.js?retry=${Date.now()}`) — a new URL is a new module, so the browser fetches for real; `force: true` is still required so the new entry replaces the registered one (it logs a benign "already registered" warning). Caching semantics are unharmed: `remoteEntry.js` is served `no-store` anyway (see [deployment.md](./deployment.md)). See [LiveRemoteBlock.tsx](../naufal-host/src/components/blocks/LiveRemoteBlock.tsx) (`retryResetRef`). Corollary: a "retry" affordance that just re-fires the same `import()` is a placebo — it can never succeed on an ES-module remote; verify retries with the network tab, not the UI state.

30. **Putting a live, interactive app on a zoom/pan canvas: two CSS-transform traps.** The canvas view (see [features.md](./features.md)) renders the whole site inside a `transform: translate() scale()` plane. Two things break under that transform and both have the same root cause — geometry APIs report _post-transform_ screen pixels while the app reasons in _pre-transform_ layout pixels. (a) **`IntersectionObserver` is unreliable through the transform** — scroll-reveal cells never reliably "intersect the viewport" on the scaled/translated plane, so they'd stay hidden. Fix: make [`useInView`](../naufal-host/src/lib/useInView.ts) canvas-aware (`return inView || canvasMode`) so everything reveals at once in canvas mode. (b) **Pointer→local coordinate mapping is off at any zoom ≠ 100%** — a draggable remote (the Svelte `SpringToy`) maps the pointer with `clientX − getBoundingClientRect().left`, but the rect is scaled while the physics use layout px, so the grab point drifts. Fix in [`SpringToy.svelte`](../naufal-lab/src/lib/SpringToy.svelte): divide the offset by the rendered ratio (`rect.width / offsetWidth`), which is `1` in normal/standalone use so nothing changes there. General rule: any component that translates `clientX/Y` into its own coordinate space must account for a transformed ancestor — divide by the measured scale rather than assuming 1:1. Pointer **events** themselves work fine through a transform (capture, bubbling, hit-testing are all in screen space); it's only the coordinate _math_ that needs the scale. The canvas's own gesture engine ([`useCanvas.ts`](../naufal-host/src/lib/useCanvas.ts)) sidesteps it by panning on incremental deltas and zooming toward a surface-local point it computes itself.

31. **Splitting a heading into per-character spans destroys its accessible name.** The hero wordmark wraps every letter in its own `<span>` (for the cursor-repel effect). Its `textContent` is still `"Naufal Yusran"`, but its **computed accessible name** isn't — the inline-block per-letter spans make the name-from-content algorithm insert separators, so a screen reader announces garbled letter-spacing and `getByRole('heading', { name: 'Naufal Yusran' })` never matches (while `locator('h1', { hasText: … })` does — `hasText` reads `textContent`, the accessible-name API doesn't). Fix: name the element explicitly with **`aria-label`** and mark the decorative letters **`aria-hidden`** (the icon-button pattern). General rule: any text fragmented into per-glyph/word elements for animation needs an explicit accessible name. This was **caught by the smoke suite on its first run** ([running-locally.md](./running-locally.md)) — the kind of thing reading the DOM never reveals; assert headings/labels by **role + accessible name**, not just visible text, and the gap surfaces.

32. **The React Compiler bans impure calls (`Date.now()`, `Math.random()`, `performance.now()`) during render.** The quality block shows "last run 3h ago", which needs the current time — but reading `Date.now()` in the render path fails the compiler's lint with `Cannot call impure function during render`. The compiler memoizes on the premise that render is a pure function of props/state, so a clock read inline in JSX (or a render-phase `const`) would be cached and silently stale. Fix: pull the impure value in a callback/effect and stash it in state — `const [loadedAt, setLoadedAt] = useState(0)` set inside the `fetch().then(...)`, then render the relative time from `loadedAt`. See [QualityBlock.tsx](../naufal-host/src/components/blocks/QualityBlock.tsx). General rule: any non-deterministic source (time, random, `performance`) must enter a component through state/refs set in an event handler or effect, never be called during render.

33. **React Testing Library component tests in jsdom need the missing browser APIs stubbed.** Vitest's `environment: 'jsdom'` provides `document`/`localStorage` but **not** `IntersectionObserver`, `ResizeObserver`, `matchMedia`, or `requestAnimationFrame` — and the host's components touch all four at mount (scroll-reveal `Cell`/`useInView` → IntersectionObserver; measured panels → ResizeObserver; the theme listener → matchMedia; the tech-stack orbit → rAF). Rendering any of them throws "X is not a constructor/function" until stubbed. Fix: a Vitest `setupFiles` ([vitest.setup.ts](../naufal-host/vitest.setup.ts)) that polyfills them (a no-op observer class, a `matchMedia` returning `matches: false`, a `setTimeout`-backed rAF) and also wires RTL's `afterEach(cleanup)` + `@testing-library/jest-dom/vitest`. General rule: jsdom is a partial DOM — component tests need a setup file stubbing the observer/visual APIs your components call on mount; keep the assertions on logic + markup and leave genuinely visual behaviour (layout, the orbit, canvas transforms) to Playwright.

34. **Playwright's `getByRole({ name })` matches as a substring; RTL's matches exactly.** When the quality block was suite cards it had two report links whose names shared a prefix — "open report" and "open report (with video)" — and `getByRole('link', { name: 'open report' })` resolved to the one exact match in **RTL** but matched **both** in the **Playwright** smoke test (Playwright does an accessible-name _substring_ match by default) → a strict-mode violation ("resolved to 2 elements"). The block is now an architecture map whose report links are named per node + reporter (`naufal-lab Vitest — open report`, the host's `host Vitest — open report` + `host Playwright — open report`, …), so each is unique by construction — but the rule stands: RTL's `name: string` is exact (use a regex for a partial match), Playwright's `name: string` is substring (pass `exact: true` for a whole-string match), so a selector that's unambiguous in one can be ambiguous in the other. (Relatedly, each link's `↗` arrow lives in an `aria-hidden` span so the accessible name stays clean for both tools.)
